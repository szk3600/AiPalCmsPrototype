Header/Footer を別ファイルに分離する設計的メリット

layout.tsx 内で完結させず、コンポーネントを独立したファイルに切り出す理由と、その恩恵について解説します。

1. 【最重要】Server と Client の境界線の明確化

Next.js App Router において、layout.tsx はデフォルトで Server Component です。一方、ヘッダーには「モバイルメニューの開閉（useState）」や「ログイン状態の監視」など、ブラウザ側でのインタラクションが必要になります。

分離しない場合:
ヘッダーに useState を使うため、layout.tsx 全体に 'use client' を付与する必要があります。その結果、レイアウト全体がクライアント側でレンダリングされ、サーバーサイドでの最適化（SEO、初期表示速度）の恩恵が薄れます。

分離する場合:
layout.tsx は Server Component のまま維持し、ヘッダーだけを 'use client' な独立コンポーネントとして読み込めます。これにより、レイアウトの骨組みはサーバー側で高速に生成しつつ、ヘッダーだけに必要なインタラクティブ性を持たせることができます。

2. ページ品質と SEO の維持

layout.tsx で Metadata を動的に生成する場合、そのレイアウト自体が Server Component であることが推奨されます。
コンポーネントを分離しておくことで、ヘッダー内の複雑なロジックがメタデータの生成処理に干渉するのを防ぎ、高い凝集度を保ったまま SEO 設定をクリーンに記述できます。

3. 保守性と「コードのノイズ」の削減

ヘッダーやフッターは、成長するにつれて以下のような要素が増えていきます：

多階層のドロップダウンメニュー

ユーザーアイコン（ログイン状態による表示切り替え）

スクロールに応じた透過・固定アニメーション

多言語対応のスイッチ

これらをすべて layout.tsx に書くと、数百行の巨大なファイルになり、肝心の「ページの全体構造」が読み取りにくくなります。ファイルを分けることは、「構造（Layout）」と「機能（Component）」を分離し、メンテナンス時の視認性を高めることにつながります。

4. 再利用性とテストの独立性

将来的に「特定のページ（LPなど）だけ別のヘッダーを使いたい」「エラーページ用の最小限のフッターが欲しい」といった要望が出た際、独立したファイルであればインポートするだけで済みます。
また、Vitest などのテストツールで「ヘッダーのメニューをクリックしたら正しく開くか」といった単体テストを書く際も、ファイルが分かれている方が圧倒的に書きやすく、高速です。

5. 今回のプロジェクトにおける指針

「凝集度」を重視される場合は、以下のようなディレクトリ構成を推奨します。
 注）componentsが増えてきたら考える。

app/
  layout.tsx (Server: 全体の骨組みとMetadata)
components/
  layout/ (レイアウト専用パーツのフォルダ)
    Header.tsx (Client: メニュー開閉などのロジック)
    Footer.tsx (Server/Client: フッター)


このように 「Layout フォルダ」 を作ることで、物理的なファイル数は増えても、役割ごとの凝集度は最大化されます。
